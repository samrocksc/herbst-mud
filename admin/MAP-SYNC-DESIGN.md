# Map State Sync & Conflict Resolution — Multi-Editor Scenarios

**Issue:** #321  
**Date:** 2026-05-03  
**Author:** Donatello 🟣

---

## The Problem

The herbst-mud admin UI (`/map`) allows authenticated admin users to edit room properties (name, description, exits) via `PUT /rooms/:id`. Currently, the last writer always wins — there is no versioning, no conflict detection, and no awareness of concurrent editors.

**Concrete scenario:**

1. Admin A opens room #12 in the editor (loads `name: "Tavern"`, `description: "A dusty inn."`)
2. Admin B opens room #12 in the editor at the same time (same stale snapshot)
3. Admin A saves first: `name: "Tavern"`, `description: "A smoky tavern."` → 200 OK
4. Admin B saves second: `name: "The Rusty Tankard"`, `description: "A dusty inn."` → 200 OK
5. Result: Admin A's description change is silently overwritten. Neither admin is aware data was lost.

The `handleSaveRoom` in `map.tsx` sends the full room body without any version check. The server `PUT /rooms/:id` unconditionally overwrites whatever is in the database. This is a classic "lost update" problem.

---

## Option A: Optimistic Locking (Version Numbers)

### How it works

Each room row gets a monotonically increasing `version` (integer). When a client reads a room, the server includes `version` in the response. When the client sends a `PUT`, it includes the `version` it read. The server compares:

- If `request.version == db.version` → apply the update, increment `version`, return 200
- If `request.version != db.version` → reject with **409 Conflict**, include the current server state in the response body

### Implementation footprint

| Layer | Change |
|-------|--------|
| **DB schema** (`server/db/schema/room.go`) | Add `field.Int("version").Default(1)` |
| **Ent generated code** | Re-run `go generate ./...` |
| **Room routes** (`server/routes/room_routes.go`) | `PUT` handler: accept `version`, compare before update, return 409 on mismatch |
| **Room model** (`server/db/room.go`) | Auto-regenerated by ent — includes `Version int` |
| **Frontend types** (`admin/src/components/map/types.ts`) | Add `version: number` to `Room` |
| **Frontend `map.tsx`** | Send `version` in `PUT` body; handle 409 — show conflict banner with options to overwrite or reload |

### Pros

- **Simple to implement** — one integer column, one comparison, one error code
- **No additional infrastructure** — no WebSocket server, no Redis, no pub/sub
- **Correct** — guarantees no silent overwrites
- **Works offline** — no persistent connection required; conflict detected at save time
- **Idempotent** — retrying a save with the right version is safe

### Cons

- **Late detection** — conflict detected only at save time, not when the other editor starts
- **Manual resolution** — user must choose to overwrite or merge; no auto-merge for complex exits changes
- **Cascade exits** — `handleSaveRoom` also updates adjacent rooms for bidirectional exits; those updates could also conflict (separate version checks needed)

## Option B: WebSocket Real-Time Sync

### How it works

Establish a WebSocket connection per admin session. When an admin opens a room for editing, broadcast "user X is editing room Y" to all connected clients. When a room is saved, broadcast the updated room to all clients so they see the latest state immediately.

### Implementation footprint

| Layer | Change |
|-------|--------|
| **Server** | Add WebSocket hub (gorilla/websocket or nhooyr.io/websocket) |
| **Routes** | New `/ws` endpoint; broadcast room-update events |
| **Frontend** | WebSocket client hook; presence indicators on room nodes; auto-refresh on remote changes |

### Pros

- **Best UX** — editors see who else is working, changes appear in real-time
- **Proactive awareness** — reduces conflicts by making them visible before they happen

### Cons

- **Heavy infrastructure** — WebSocket server, connection management, reconnection logic, heartbeat, scaling concerns
- **Complex frontend** — new hook, new state synchronization, race conditions between WS messages and REST saves
- **Partial solution** — does not *prevent* conflicts, only makes them visible; still needs a conflict strategy (likely optimistic locking as a fallback)
- **Deployment complexity** — load balancer sticky sessions or Redis pub/sub for multi-instance

## Option C: Pessimistic Locking (Room-Level Locks)

### How it works

When an admin opens a room for editing, the server locks it. Other admins see the room as "locked by User X" and cannot edit until the lock is released (on save, cancel, or timeout).

### Implementation footprint

| Layer | Change |
|-------|--------|
| **Server** | Lock table or Redis key: `room_locks[room_id] = { user_id, timestamp }` |
| **Routes** | `PUT` checks lock ownership; new `POST /rooms/:id/lock` and `DELETE /rooms/:id/lock` endpoints |
| **Frontend** | Show lock status on room nodes; disable edit button for locked rooms; heartbeat to extend lock |

### Pros

- **Simplest mental model** — "I'm editing this, you can't"
- **No lost updates** — conflicts prevented entirely

### Cons

- **Lock staleness** — what if the editor's browser crashes? Lock stays forever until timeout. Requires heartbeat.
- **Reduced parallelism** — even if two editors are changing different fields, one is blocked
- **State management** — lock acquisition/release must be atomic; need a cleanup mechanism for stale locks
- **Network partitions** — if the lock endpoint is unreachable, no one can edit
- **UX friction** — "Room is locked" is frustrating for a small-team scenario where the likely solution is "just ask the other person"

---

## Recommendation: Option A — Optimistic Locking

**Rationale:**

1. **Scale of the problem** — herbst-mud is a small-team MUD engine. Simultaneous edits are rare, not constant. Optimistic locking handles the edge case without over-engineering.

2. **Incremental adoption** — optimistic locking can ship independently. Options B and C can be layered on later (WebSocket for awareness, locking for high-contention rooms) without breaking the version contract.

3. **Correctness floor** — the current situation (silent last-write-wins) is a data integrity bug. Optimistic locking fixes it with minimal code. The 409 response gives the UI a clear signal to show the user.

4. **Exit cascade handling** — the exit-synchronization logic in `handleSaveRoom` (updating bidirectional exits on adjacent rooms) is already complex. Adding version checks to those PUTs is straightforward; WebSocket would require coordination across multiple rooms in real-time.

5. **No new infrastructure** — no Redis, no WebSocket library, no connection management. One column, one check, one error code.

### Implementation plan

1. Add `version` field to the Room ent schema
2. Regenerate ent code (`go generate`)
3. Update `PUT /rooms/:id` to accept and validate `version`; return 409 on mismatch with current room state
4. Update `GET /rooms` and `GET /rooms/:id` to include `version` in responses
5. Add `version` to the frontend `Room` type
6. Update `handleSaveRoom` to send `version`; handle 409 with a conflict banner
7. Handle exit-cascade PUTs similarly (include version of the target room being modified)

### Conflict resolution UX

When a 409 is received, the frontend should:

1. Show a banner: "This room was modified by another editor while you were editing."
2. Display two options:
   - **Reload** — discard local changes, reload the room from server
   - **Overwrite** — force-save with the current server version (increment version number and retry)
3. The 409 response body includes the current server state so the user can compare

### Exit cascade considerations

`handleSaveRoom` performs up to N additional PUTs for bidirectional exit updates. Each of these target rooms also has a version. For the initial implementation:

- If a cascade PUT gets a 409, surface the error to the user with the specific room that conflicted
- The user can either retry the whole save or reload and re-edit
- This is acceptable because exit cascade conflicts are even rarer than direct room conflicts

### Future enhancements (out of scope for #321)

- WebSocket presence layer (Option B) for real-time "who's editing" indicators
- Operational transforms for field-level merge (auto-merge name from A, description from B)
- Lock timeout configuration in admin settings
